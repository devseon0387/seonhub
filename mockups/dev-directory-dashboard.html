세 시안 비교

A · Launcher Grid

  • 전체 조망 가장 빠름 (26개 한 화면)
  • 앱 아이콘 느낌 · iOS 홈 유사
  • 상태는 작은 점 하나로만
  • "빠르게 열기" 단축 UX
  • 단점: 메타정보 적음

B · Project Cards

  • 설명·기술·포트 한 카드에 요약
  • 중간 밀도 · 한 화면 9~12개
  • 퀵액션(에디터/브라우저/터미널)
  • 상태 뱃지 가독성 좋음
  • 단점: 그리드 때문에 정렬·필터 한계

C · Data Table

  • 정보 밀도 최대 · 한 행 한 프로젝트
  • 정렬/필터 가능 컬럼 많음
  • 커밋 수·사이즈·브랜치 등 기술 정보
  • 개발자 친화적
  • 단점: 시각적 인상 약함
내 추천: 26개 정도 규모 + 자주 접속하는 건 몇 개 안 됨 → A와 C의 하이브리드가 최적. 기본 화면은 Launcher Grid(A)로 시각적 스캔, "테이블 뷰" 토글을 제공해서 정렬/필터/기술 세부는 C로. 카드(B)는 중간 지대라 너무 많은 프로젝트엔 부족, 몇 개에선 과함.

구현 방향:
  • → 백엔드: fs.readdir("~/Desktop/Dev") + package.json 파싱 + git status
  • → 상태 실시간: 포트 lsof로 running 감지 (주기적 polling 10~30초)
  • → 메타 저장: 앱 내 DB에 각 프로젝트 별명·태그·숨김 여부 저장 (디렉토리 없는 건 아카이브)
  • → 퀵액션: VS Code 열기 (vscode://file/...), 브라우저 열기, iTerm/Warp 열기 (앱 스킴)
  • → Dev 경로는 설정에서 변경 가능하게 (~/Desktop/Dev 기본, 여러 경로 추가 가능)

데이터 모델 초안

DevProject {
  id: uuid
  path: string                   // 절대경로, fs 스캔으로 자동 발견
  name: string                   // 기본: 디렉토리명, 사용자가 별명 지정 가능
  description: string | null     // 사용자 메모 또는 package.json의 description
  category: 'active' | 'production' | 'experiment' | 'utility' | 'archive'
  tags: string[]                 // 사용자 정의
  colorGradient: [string, string]  // 아이콘 색상
  iconInitials: string           // "SH", "P", "도"

  // fs 감지 (매 스캔마다 갱신)
  packageJson: object | null     // Next.js, Python requirements.txt 등
  devPort: number | null         // package.json scripts에서 파싱 (-p 플래그)
  techStack: string[]            // 자동 감지: Next.js, Python, C++, etc.
  lastModified: timestamp
  sizeOnDisk: number

  // git (실시간)
  gitBranch: string | null
  gitUncommitted: boolean
  gitCommitsThisWeek: number

  // 런타임 (polling)
  isRunning: boolean             // lsof로 devPort 점유 확인
  runningPid: number | null

  // 사용자 상태
  isPinned: boolean
  isHidden: boolean
  lastOpenedAt: timestamp
}
